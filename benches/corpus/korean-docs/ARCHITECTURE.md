# DocuSearch 시스템 아키텍처 설계

DocuSearch는 대규모 문서 검색 시스템으로, 한국어 자연어 처리와 벡터 검색을 결합한 하이브리드 검색 엔진을 핵심으로 설계되었습니다.

## 전체 시스템 구성

시스템은 다음 네 가지 계층으로 구성됩니다.

1. **프레젠테이션 계층** -- React 기반 웹 UI와 REST API 게이트웨이.
2. **검색 엔진 계층** -- Elasticsearch 전문 검색과 벡터 검색을 통합한 쿼리 실행부.
3. **문서 처리 계층** -- 파일 업로드, 텍스트 추출, 청킹, 임베딩 생성을 담당.
4. **저장소 계층** -- PostgreSQL (메타데이터), Elasticsearch (인덱스), S3 (원본 파일).

## 검색 시스템

### 전문 검색 (Full-Text Search)

Elasticsearch를 기본 검색 엔진으로 사용합니다. 한국어 처리를 위해 Nori 형태소 분석기를 적용하며, 다음과 같은 분석 파이프라인을 구성합니다:

- **문서 분석기**: `nori_tokenizer` + `nori_part_of_speech` 필터 (조사, 어미 제거) + `nori_readingform` 필터 (한자-한글 변환).
- **검색 분석기**: 동일한 Nori 파이프라인에 `synonym` 필터를 추가하여 동의어 확장 지원.
- **필드 가중치**: 제목(3.0), 본문(1.0), 태그(2.0), 작성자(1.5) 가중치를 적용한 BM25 점수 계산.

### 벡터 검색 (Vector Search)

문서의 의미적 유사도를 검색하기 위해 임베딩 모델을 사용합니다.

- **임베딩 모델**: `bge-m3` 다국어 임베딩 모델 (1024차원).
- **청킹 전략**: 문서를 512 토큰 단위로 분할하며, 문장 경계에서만 청크를 나눔.
- **인덱스**: Elasticsearch의 `dense_vector` 필드 타입과 `kNN` 검색을 활용.
- **거리 척도**: 코사인 유사도 (cosine similarity).

### 하이브리드 검색 결합

전문 검색 점수와 벡터 검색 점수를 선형 결합하여 최종 순위를 결정합니다.

- 기본 가중치: BM25 점수 60%, 벡터 유사도 40%.
- 관리자가 쿼리 유형에 따라 가중치를 조정 가능.
- 검색 결과 하이라이팅은 BM25 매칭 결과를 기반으로 생성.

## 데이터 처리

### 문서 수집 파이프라인

1. 사용자가 파일을 업로드하면 S3에 원본 파일을 저장.
2. 비동기 워커가 파일을 polling하여 Apache Tika로 텍스트 추출.
3. 추출된 텍스트를 청킹하고 각 청크에 임베딩을 생성.
4. 메타데이터(제목, 작성자, 작성일, 태그)를 PostgreSQL에 저장.
5. 청크 텍스트와 임베딩 벡터를 Elasticsearch에 인덱싱.
6. 인덱싱 완료 후 프론트엔드에 WebSocket으로 알림.

### 임베딩 모델 서빙

임베딩 모델은 FastAPI 기반 별도 서비스로 배포됩니다.

- GPU 인스턴스에서 모델을 로드하여 REST API로 임베딩 요청 처리.
- 배치 처리 지원: 최대 32개 청크를 한 번에 임베딩.
- 요청 큐를 통해 트래픽 스파이크 시 자동 버퍼링.
- 모델 교체 시 무중단 롤링 배포 지원.

## API 설계

RESTful API는 다음과 같은 엔드포인트를 제공합니다:

- `POST /api/v1/documents` -- 문서 업로드 및 인덱싱 트리거.
- `GET /api/v1/search?q={query}&filters={...}` -- 하이브리드 검색 실행.
- `GET /api/v1/documents/{id}` -- 문서 상세 조회.
- `PUT /api/v1/documents/{id}/permissions` -- 문서 권한 수정.
- `GET /api/v1/suggestions?q={prefix}` -- 검색어 자동완성.

모든 API 응답은 JSON 형식이며, 페이지네이션은 `offset`과 `limit` 파라미터를 사용합니다. 오류 응답은 RFC 7807 (Problem Details) 형식을 따릅니다.
